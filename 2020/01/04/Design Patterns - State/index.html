Design Patterns - State

 2020-01-04  2020-01-04
Design Patterns / Behavioral
9 minutes membaca (Sekitar 1291 kata)

本文将介绍设计模式中的状态模式（State）。

定义

该模式允许对象在内部状态改变时改变它的行为，对象看起来好像修改了他的类。即将状态封装为独立的类，并将动作委托到当前对象。

实现

通过抽象类定义状态接口，以及共有方法。然后由具体子类定义具体状态及在该状态下的动作方式。

状态模式 VS 策略模式

从 UML 类图结构来看，状态模式和策略模式简直完全一样，它们之间的差别在于它们的“意图”。

  • 状态模式：我们将一群行为封装在状态对象中，context 的行为岁时可以委托到那些状态对象中的一个。随着时间流逝，当前状态在状态对象集合中游走改变，以反映出 context 内部的状态。因此， context 的行为也会跟着改变。但是 context 的客户对于状态对象了解不多，甚至浑然不觉。我们将状态模式想成是不用在 context 中放置许多条件判断的替代方案。
  • 策略模式：客户通常主动指定 context 所要组合的策略对象是哪一个。现在，固然策略模式让我们巨头弹性，能够在运行时改变策略，但对于某个 context 对象来说，通常会有一个最适合的策略对象。一般来说，策略模式是除了继承之外的一种弹性替代方案。如果你使用继承定义了一个类的行为，你被这个行为困住，甚至要修改它都很难，有了策略模式，就可以通过组合不同对象来改变行为。
